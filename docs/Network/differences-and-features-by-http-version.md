## HTTP/1.0

- 1996년에 등장한 초기 버전
- 요청마다 새로운 TCP 연결을 생성하고 응답 후 즉시 연결을 종료하는 방식
- 매 요청마다 3-way handshake 과정이 필요하여 여러 리소스를 가져올 때 성능 저하 발생

## HTTP/1.1

1999년에 등장했으며 웹의 성능을 크게 개선함

**특징**

- Persistent Connection: 하나의 TCP 연결로 여러 요청/응답을 처리하며, keep-alive 헤더를 통해 연결 유지
- Pipelining: 이전 요청의 응답을 기다리지 않고 여러 요청을 연속으로 보낼 수 있음. 그러나 응답은 여전히 순서대로 받아야 함 (FIFO)
- Host 헤더: 하나의 IP에서 여러 도메인을 호스팅하는 가상 호스팅이 가능해짐
- 캐시 제어 강화: Cache-Control, ETag 등의 헤더로 더 정교한 캐싱이 가능해짐

**한계점**

- Head-of-Line Blocking: 파이프라이닝에서 첫 번째 응답이 지연되면 뒤의 모든 응답도 대기
- 무거운 헤더: 매 요청마다 중복된 헤더 정보를 텍스트로 전송
- 우선순위 처리 불가: 중요한 리소스를 먼저 받을 방법이 없음

## HTTP/2

2015년에 등장했으며 SPDY 프로토콜을 기반으로 만들어졌으며, HTTP/1.x의 시간 지연 문제를 해결

- SPDY: 웹 콘텐츠를 전송할 목적으로 구글이 개발한 비표준 개방형 네트워크 프로토콜

**특징**

1. Binary Framing Layer
   - 텍스트 기반이었던 HTTP/1.x와 달리 바이너리 프레임으로 데이터를 전송
   - 하나의 연결 안에서 여러 스트림이 병렬로 동작하며, 각 스트림은 메시지를 구성하는 프레임들로 이루어짐
2. Multiplexing
   - 하나의 TCP 연결에서 여러 요청과 응답을 동시에 처리
   - 각 요청은 고유한 스트림 ID를 가지며, 응답 순서에 상관없이 병렬 처리
   - HTTP/1.1의 Head of Line Blocking 해결
3. Header Compression
   - 중복되는 헤더 정보를 압축하고, 이전에 전송한 헤더를 참조하는 방식으로 헤더 크기를 줄임
   - 쿠키처럼 큰 헤더가 반복될 때 효과적
4. Server Push
   - 클라이언트가 요청하지 않아도 서버가 필요할 것으로 예상되는 리소스를 미리 보냄
   - 예를들어 HTML을 요청하면 서버가 CSS와 JS 파일을 함께 푸시
5. Stream Prioritization
   - 각 스트림에 우선순위를 부여하여 중요한 리소스를 먼저 전송
   - 의존성 트리를 구성해 리소스 로딩 순서를 최적화

**한계점**

- TCP 레벨의 Head-of-Line Blocking이 존재하여, 패킷 손실이 발생하면 TCP 재전송 메커니즘 때문에 모든 스트림이 영향을 받음

## HTTP/3

- 2022년에 표준화된 HTTP/3는 TCP 대신 QUIC 프로토콜을 사용

**특징**

1. QUIC over UDP
   - TCP가 아닌 UDP 기반의 QUIC 프로토콜 사용
   - QUIC: 신뢰성, 혼잡 제어, 연결 관리 등 TCP 기능을 UDP 위에 구현하면서도 더 효율적으로 설계
2. 완전한 Multiplexing
   - TCP 레벨의 Head-of-Line Blocking이 완전히 해결
   - 각 스트림이 독립적으로 동작하므로 하나의 스트림에서 패킷 손실이 발생해도 다른 스트림에는 영향을 주지 않음
3. 0-RTT Connection Establishment
   - 이전에 연결했던 서버라면 TLS 핸드셰이크 없이 즉시 데이터 전송
4. Connection Migration
   - 연결 ID를 사용하여 IP 주소나 포트가 바뀌어도 연결 유지
   - 모바일 환경에서 Wi-Fi와 셀룰러 네트워크 간 전환 시에도 끊김 없이 연결 지속
5. 향상된 보안
   - TLS 1.3이 필수로 내장되어 있어 암호화가 기본이며, QUIC 프로토콜 자체에 보안이 통합됨
